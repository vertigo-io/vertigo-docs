# Card

A card allows representing a business entity in a highly visual way within a collection.
A card has several information areas: a header, a middle content area, and a footer.

This type of representation is very useful when your application will be used both on mobile and on a desktop computer, because cards can automatically rearrange on the screen depending on available space.

# Best Practices

- The width of a card is an important choice; it should be determined based on the content inside, but also by the number of elements you want to display side by side, and therefore by the context in which it will be placed
- A card should contain a notable graphic element: an image, a chart, and should not consist solely of text and buttons
- Limit the number of available actions on a card (<3); for complex actions, prefer navigation to a dedicated screen
- Use shadows on cards to better define their boundaries, and optionally a subtle animation on hover to guide the user
- Prefer showing an additional page via a "See more" type button
